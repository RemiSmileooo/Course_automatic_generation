# 需求文档：LLM 自主生成幻灯片 HTML

## 简介

当前项目把课程文稿转成「带配音/字幕的 MP4 视频 + PPTX」。页面渲染目前要么是 Pillow 位图绘制，要么是代码套固定 HTML 模板——版式有限、容易单调。

本功能把**每页的视觉设计权交给 LLM**：给 LLM 该页内容 + 一套「设计系统规范」，由 LLM 自主决定这一页用什么布局（分栏 / 网格 / 时间线 / 大字宣言 / 对照 / 卡片流等）并直接产出整页 HTML。目标是「千页千面、不单调」，同时通过设计系统约束保证整套 PPT 视觉统一。

关键原则：
- **LLM 只负责"页面内容层"的设计**；**字幕层由代码统一叠加**，以保证与 TTS 音频逐句精确对齐（这条线不可越界）。
- **设计系统约束视觉气质**（配色 token、字体、圆角阴影、字幕安全区、尺寸），**布局结构完全放开**给 LLM，以兼顾「统一」与「不雷同」。
- **必须有兜底**：LLM 产物校验失败时，回退到固定模板渲染，保证永远能出片。

设计系统以 `design_preview.html`（浅色暖橙风）为基础抽象而来。

本功能聚焦渲染层（LLM 生成 HTML → 截图 → 字幕帧）；TTS、字幕切分、视频合成沿用现有契约，不改动。

## 术语表

| 术语 | 含义 |
|---|---|
| 设计系统 | 一套统一的 CSS 变量(配色 token)、字体、圆角/阴影风格、字幕安全区与画布尺寸规范，约束所有页面的视觉气质 |
| 内容层 | 页面中承载课程内容的部分（标题、要点、表格、图形化布局），由 LLM 设计 |
| 字幕层 | 画面底部随口播逐句滚动的字幕条，由代码叠加，不交给 LLM |
| 渲染契约 | 每页生成一张干净底图(写入 `slide.base_image`)，每条 Caption 生成一帧 PNG(写入 `caption.frame_path`)，尺寸 = 视频分辨率 |
| 字幕安全区 | 画面底部预留给字幕条的区域，LLM 设计内容时不得占用，避免内容与字幕重叠 |
| 兜底模板 | 代码内置的固定 HTML 模板，当 LLM 产物校验失败时使用 |
| Slide / Segment / Caption | 现有数据模型：页 / 讲解片段 / 逐句字幕 |

## 需求

### 需求 1：LLM 自主生成每页内容层 HTML

**用户故事：** 作为使用者，我希望每页的版式由 LLM 根据内容自主设计，使每次生成的 PPT 版式多样、不单调。

#### 验收标准

1. WHEN 渲染某页 THEN 系统 SHALL 调用 LLM，输入该页结构化内容(标题、要点/表格、教学角色等)与设计系统规范，获取该页「内容层」HTML 片段。
2. THE LLM SHALL 自主决定该页的布局结构（如分栏、网格、时间线、大字强调、对照、卡片流等），不受固定模板限制。
3. THE LLM 生成的 HTML SHALL 只包含内容层，不得包含字幕条或任何与逐句字幕相关的元素。
4. WHILE 同一门课的多页生成过程中 THE 系统 SHALL 通过提示鼓励相邻页采用不同布局手法，以避免雷同。
5. THE 系统 SHALL 在调用 LLM 时提供该页的内容数据，使 HTML 中的文字内容忠实于课程内容，不臆造事实。

### 需求 2：设计系统约束（视觉统一）

**用户故事：** 作为使用者，我希望尽管每页布局不同，但整套 PPT 看起来像同一套设计，风格统一不乱跳。

#### 验收标准

1. THE 系统 SHALL 向 LLM 提供一套设计系统规范，包含：配色 token（背景、标题、正文、强调色）、字体、圆角/阴影风格、画布尺寸、字幕安全区。
2. THE 设计系统 SHALL 以浅色暖橙风（基于 design_preview.html）为基础。
3. WHEN LLM 生成 HTML THEN 颜色、字体 SHALL 来自设计系统提供的变量/类，而非随意硬编码新风格。
4. THE 生成页面 SHALL 固定为配置的视频分辨率（VIDEO_WIDTH × VIDEO_HEIGHT）。
5. THE 系统 SHALL 使用项目探测到的中文字体渲染，保证中文不缺字、不回退成方框。
6. THE LLM SHALL 在内容布局中预留底部字幕安全区，不得在该区域放置内容元素。

### 需求 3：字幕层由代码控制并与 TTS 对齐

**用户故事：** 作为系统维护者，我要求字幕与配音逐句精确对齐，这一点不能因为引入 LLM 设计而被破坏。

#### 验收标准

1. THE 字幕切分与计时 SHALL 继续由代码完成（复用现有 `build_captions`），不交给 LLM。
2. WHEN 为某页生成逐句字幕帧 THEN 系统 SHALL 在 LLM 生成的同一页内容层之上，由代码叠加字幕条，再截图。
3. THE 每条 Caption SHALL 对应一张完整帧 PNG，写入 `caption.frame_path`，命名规则与现有一致。
4. THE 字幕条样式 SHALL 由代码统一控制，并保证在设计系统配色下清晰可读。
5. THE TTS、字幕计时、视频合成阶段 SHALL 不因本功能改动其行为。

### 需求 4：校验与兜底（永远能出片）

**用户故事：** 作为系统维护者，我希望 LLM 偶尔产出异常 HTML 时，系统能自动兜底，绝不让整条流程中断或产出破损画面。

#### 验收标准

1. WHEN 收到 LLM 生成的 HTML THEN 系统 SHALL 校验其基本有效性（非空、可被浏览器加载、关键结构存在）。
2. IF LLM 调用失败、超时、返回空或校验不通过 THEN 系统 SHALL 回退到固定兜底模板渲染该页。
3. WHERE 检测到内容明显溢出画面或越界进入字幕安全区 THE 系统 SHALL 视为校验失败并回退兜底模板。
4. WHEN 浏览器/Playwright 不可用 THEN 系统 SHALL 回退到 Pillow 渲染（沿用现有机制），保证视频仍可生成。
5. WHERE 未配置 LLM(无 API Key) THE 系统 SHALL 直接使用固定兜底模板，不中断流程。
6. THE 系统 SHALL 记录每页实际使用的渲染方式（LLM / 兜底模板 / Pillow）及回退原因，便于排查。

### 需求 5：渲染契约与产物保留

**用户故事：** 作为系统维护者，我要求新渲染方式产出的中间产物结构与现有一致，便于调试与下游复用。

#### 验收标准

1. THE 每页 SHALL 生成一张干净底图(无字幕)，写入 `slide.base_image`。
2. THE 系统 SHALL 把每页 LLM 生成的 HTML 源码落盘（如 `slides/slide_XX.html`），便于查看与调试。
3. THE 输出 PNG 分辨率 SHALL 等于配置的视频分辨率。
4. THE PPTX 导出 SHALL 能继续工作（沿用现有逻辑或使用底图，二选一在设计阶段确定）。

### 需求 6：性能与成本可控

**用户故事：** 作为使用者，我希望引入 LLM 设计后，生成时间和成本仍在可接受范围。

#### 验收标准

1. THE 系统 SHALL 对每页仅调用一次 LLM 生成 HTML（不做多轮往返）。
2. WHERE LLM 生成失败 THE 系统 SHALL 直接兜底，不无限重试（最多重试次数在设计阶段确定，建议 ≤1）。
3. THE 系统 SHOULD 在进度回调中反映「LLM 设计页面中」的阶段，避免界面长时间无反馈。

## 关键决策（已确认）

- **D-1（OQ-1）LLM 输出格式**：LLM **直接返回整页内容层 HTML 片段**（纯文本），不走结构化 JSON 再组装。追求最强自主性与多样性。
- **D-2（OQ-3）校验严格度**：**仅做语法/尺寸等轻量校验**（非空、能加载、根元素存在、无明显语法破损），不做浏览器逐元素溢出实测。校验不过即回退兜底模板。
- **D-3（OQ-4）缓存**：**启用缓存**——同一页内容(指纹)缓存 LLM 生成结果，重试/调试时不重复请求。
- **D-4（OQ-5）调用通道**：复用现有 OpenAI 兼容接口；**HTML 以纯文本返回**（不包在 JSON 字段里），代码做必要清洗（去掉 ```html 围栏等）。
- **D-5（OQ-2 / 架构）已定：采用架构 Y（LLM 端到端）**。给 LLM 一段原始文案 + 设计系统规范，由 LLM 一次性决定：分几页、每页内容、每页内容层 HTML 设计、每页口播稿。代码负责：把 LLM 产物解析成内部数据模型、字幕切分对齐、TTS、合成、以及校验与兜底。

## 架构 Y 的补充约束（因端到端风险更高而增加）

- **页数控制**：系统 SHALL 在提示中约束分页数量的合理范围（如 6~12 页，随文案长度），避免页数失控。
- **口播与画面来源一致**：每页的口播稿与该页 HTML 内容由同一次 LLM 生成，二者 SHALL 语义对应（讲的就是这页展示的内容）。
- **内容忠实度**：LLM SHALL 基于给定文案组织内容，不臆造文案中不存在的事实/数据。
- **字幕仍由代码切分**：无论口播稿来自 LLM，字幕的切句与计时 SHALL 仍由代码 `build_captions` 完成（需求 3 不变）。
- **逐页兜底**：任一页 HTML 校验失败 SHALL 单独回退兜底模板，不影响其它页；若整次 LLM 端到端调用失败，SHALL 回退到「现有规则拆课 + 兜底模板」路径，保证出片。

> 架构已确认，可进入设计阶段。

## 原始开放问题（保留记录）

- **OQ-1：LLM 输出格式**（已定 D-1：直接 HTML）——让 LLM 直接返回 HTML 片段，还是返回结构化 JSON(布局类型 + 内容槽)再由代码组装？前者自主性最强但校验难，后者更可控但限制创造力。
- **OQ-2：设计系统交付给 LLM 的形式**——是把整段 base CSS + 类清单塞进 prompt，还是给一份"可用类/变量速查表"？影响 token 成本与遵从度。
- **OQ-3：溢出检测的严格程度**——仅检测语法/尺寸，还是用浏览器测量元素是否超出画布/进入字幕安全区？后者更稳但更慢。
- **OQ-4：是否缓存**——同一页内容是否缓存 LLM 结果，避免重复生成（如重试/调试时）。
- **OQ-5：用哪个 LLM 调用通道**——复用现有 `llm._chat_json` 的 OpenAI 兼容接口，HTML 用纯文本返回还是包在 JSON 字段里。

# 增补需求：设计-预览-对话修改-确认 工作流（第二阶段）

## 增补简介

把渲染从"一条龙跑完"拆成两个阶段，中间加人工关卡：
1. **设计阶段**：输入文案 → LLM 生成整套 PPT 的 HTML 设计（不做 TTS/视频，秒级）。
2. **预览与对话修改**：前端三区界面（HTML 源代码 + 实时渲染预览 + 对话框）。用户与后端 LLM 直接对话微调（单页改 / 整篇重改），满意为止。
3. **确认生产**：用户点"确认生成视频" → 才走 TTS + 字幕 + 合成。

价值：设计满意后再花时间做配音和编码，不浪费；对话式修改最自然灵活。

## 增补需求

### 需求 7：设计阶段（只生成设计，不生产）

#### 验收标准
1. WHEN 用户提交文案并选择"设计" THEN 系统 SHALL 调用 LLM 生成整套课程的逐页 HTML 设计与口播稿，但 SHALL NOT 触发 TTS 与视频合成。
2. THE 设计结果 SHALL 以会话(session)形式保存在后端，含每页 HTML、口播稿、页序。
3. THE 设计阶段 SHALL 在数秒级返回（不含截图/配音）。
4. THE 系统 SHALL 强化 LLM 设计能力（丰富的设计系统组件 + 多样化版式示例 + 内容→版式映射 + 创意引导），使各页版式多样、不雷同、不退化为单一卡片墙。

### 需求 8：前端预览界面（代码 + 渲染）

#### 验收标准
1. THE 前端 SHALL 提供页签在各页之间切换。
2. THE 前端 SHALL 展示当前页的 HTML 源代码（可滚动、可读）。
3. THE 前端 SHALL 同步展示当前页 HTML 的实时渲染预览（所见即所得）。
4. WHEN 设计被修改 THEN 源代码与渲染预览 SHALL 同步刷新。
5. THE 前端 SHALL 提供"确认，生成视频"入口。

### 需求 9：对话式修改

#### 验收标准
1. THE 前端 SHALL 提供对话框，用户用自然语言提出修改诉求。
2. WHEN 用户发送修改指令 THEN 系统 SHALL 把"当前设计 + 指令"交给 LLM，返回更新后的设计。
3. THE 修改 SHALL 支持单页调整（如"第3页改成时间线"）与整套调整（如"全部配色更淡"）。
4. THE 系统 SHALL 在修改后保留会话上下文，支持多轮连续修改。
5. IF 某次修改 LLM 失败或产物非法 THEN 系统 SHALL 保留上一版设计并提示，不破坏已有结果。

### 需求 10：确认生产

#### 验收标准
1. WHEN 用户点击"确认，生成视频" THEN 系统 SHALL 以当前会话中的设计(每页 HTML + 口播稿)走 TTS → 字幕 → 渲染截图 → 合成。
2. THE 生产阶段 SHALL 复用已确认的 HTML，不再重新生成设计。
3. THE 字幕 SHALL 仍由代码逐句切分并与 TTS 对齐（需求 3 红线不变）。
4. THE 兜底链 SHALL 保持：单页 HTML 非法→兜底模板；浏览器不可用→Pillow。

## 增补开放问题
- **OQ-A：会话存储**——进程内存(简单，重启丢失) vs 落盘(可恢复)。建议先进程内存 + 落盘 JSON 备份。
- **OQ-B：预览渲染**——前端直接用 iframe/srcdoc 渲染 HTML（快、所见即所得）即可；最终出片仍由后端 Playwright 截图。
- **OQ-C：旧的一条龙模式是否保留**——保留 `SLIDE_RENDERER` 直接出片路径作为兼容，新工作流作为 Web 默认体验。

# 增补需求：逐要点高亮（模式 A，第三阶段）

## 增补简介
把每页从"一张静态图"升级为"base + 若干高亮态"的讲解动画：讲到第 N 个要点时高亮该要点（模式 A，高亮段=口播段=音频段，天然对齐语音）。高亮的视觉样式由 LLM 每页自行设计；高亮的位置与时机由结构（data-hl 标记 + 分段口播）决定。版面在各高亮态之间保持稳定（同一 DOM 仅切换 class），避免视频抖动。

## 增补需求

### 需求 11：分段口播与高亮标记（LLM 输出）
1. THE LLM 为每页 SHALL 输出一个 base HTML（不含高亮）与一个分段列表 steps；每个 step 含 focus(高亮目标编号，0 表示不高亮/开场)与 narration(该段口播稿)。
2. THE LLM SHALL 在 base HTML 中为可高亮的部分打标记 data-hl="1".."N"，编号与 steps 的 focus 对应。
3. THE LLM SHALL 在该页 HTML 内自行设计高亮样式 .hl-on(被高亮)与 .hl-dim(其余弱化)；未提供时由系统注入默认聚焦样式兜底。
4. THE 每页讲解节奏 SHALL 为"开场(可选) + 逐要点各一段"，口播分段化。

### 需求 12：高亮帧渲染与对齐
1. THE 系统 SHALL 先渲染无高亮 base 帧写入 slide.base_image。
2. WHEN 渲染某个 step（focus>0）THEN 系统 SHALL 给 [data-hl=focus] 加 .hl-on、其余 [data-hl] 加 .hl-dim 后截图；focus<=0 则渲染无高亮帧。
3. THE 各高亮态 SHALL 基于同一 base DOM 仅切换 class，版面不得跳动。
4. THE 每个 step SHALL 对应其 narration 的音频；字幕仍由代码按该段切分并对齐（红线不变）。
5. THE 高亮 class 的注入 SHALL 由系统在渲染包装层完成（可用系统脚本），不要求 LLM 输出脚本。

### 需求 13：高亮可对话微调
1. THE 用户 SHALL 能通过对话调整高亮（如"第3页第2个要点改成方框""高亮时背景虚化""不要弱化其它部分"），由 revise 重新生成 base+steps+样式。
2. THE 前端预览 SHALL 能查看各 step 的高亮态（不仅 base）。

## 增补决策
- 采用"单 base + class 切换"而非"每态独立整页 HTML"，以保证版面稳定、不抖动。
- focus 编号复用 Segment.bullet_index 存储；step.narration 存 Segment.script。
